## ✅ 체크포인트 ⭐🔥

1. **TS는 JS를 바꾸지 않는다.**

- 컴파일 단계에서 **타입 검사**를 하고, 결과물은 **JavaScript로 변환**된다.
- 실행은 결국 JS(브라우저/Node)가 한다.

2. **TS는 런타임 버그를 0으로 만들지 않는다.**

- TS가 잡는 건 “타입 실수”의 상당 부분.
- API 응답/사용자 입력처럼 런타임 데이터는 별도의 **검증(Validation)**이 필요할 수 있다.

3. **안전함(정적 타입) + 편함(추론)**을 함께 노린다.

- 모든 타입을 다 적게 강요하지 않는다(점진적 타이핑).
- 대신 “중요한 경계”부터 타입을 잡는 게 효율적이다.

4. **현업 기본 4종 세트**

- `type` / `interface` / `generics` + **narrowing(좁히기)**

## 1) TypeScript는 뭐냐?

TypeScript는 **JavaScript에 “타입 기능”을 추가한 확장판**입니다.

- JS 문법은 그대로 사용 가능
- 필요한 부분에 타입을 얹어 **실행 전에 오류를 줄이는 방향**으로 개발

---

## 2) 왜 TS가 필요해졌을까?

JavaScript는 **동적 타입**이라, 값에 따라 타입이 바뀌고 오류가 **런타임**에 터질 수 있습니다.

TypeScript의 목적은 단순합니다.

- **실행 전에** 말이 안 되는 코드를 최대한 잡아서
- **서비스에서 터지는 런타임 오류 확률**을 낮추는 것

---

## 3) TS는 어떻게 동작할까? ⚙️

TypeScript는 “실행 언어”라기보다 **검사 + 변환(컴파일) 언어**로 이해하면 쉽습니다.

1. TS 코드 작성
2. 컴파일 단계에서 **Type Checking(타입 검사)**
3. 통과하면 **JS로 변환**
4. 브라우저/Node가 JS 실행
